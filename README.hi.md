एक नियंत्रित, दोहरे-साक्षी वाला, नियतात्मक रजिस्ट्री, जिसमें दोहराने योग्य इतिहास और वैकल्पिक बाहरी प्रमाणन की सुविधा है।

---

## रेजिस्ट्रम क्या है

रेजिस्ट्रम एक **संरचनात्मक रजिस्ट्री** है, जिसका उद्देश्य विकसित होने वाली प्रणालियों में पठनीयता बनाए रखना है।

यह स्पष्ट सीमाओं के तहत अवस्था परिवर्तनों को रिकॉर्ड, मान्य और व्यवस्थित करता है, ताकि संरचना को एंट्रॉपी (अव्यवस्था) के बढ़ने पर भी समझने योग्य बनाए रखा जा सके।

| गुण | अर्थ |
| ---------- | --------- |
| संरचनात्मक | यह अर्थ पर नहीं, बल्कि रूप पर काम करता है। |
| नियतात्मक | समान इनपुट → समान आउटपुट, हमेशा। |
| त्रुटि-रोधी (फेल-क्लोज्ड) | अमान्य इनपुट के कारण गंभीर विफलता होती है, आंशिक सुधार नहीं। |
| दोहराने योग्य | ऐतिहासिक निर्णयों को समान परिणामों के साथ फिर से निष्पादित किया जा सकता है। |
| गैर-एजेंटिक | यह कभी भी कार्य नहीं करता, निर्णय नहीं लेता और न ही अनुकूलन करता है। |

**रेजिस्ट्रम यह सुनिश्चित करता है कि परिवर्तन पठनीय रहें।**

---

## रेजिस्ट्रम क्या नहीं है

रेजिस्ट्रम स्पष्ट रूप से **यह नहीं है**:

- एक अनुकूलक
- एक एजेंट
- एक निर्णय लेने वाला
- एक नियंत्रक
- एक सिफारिशकर्ता
- एक बुद्धि
- अनुकूलनीय या सीखने वाला
- स्व-सुधार करने वाला

यह कभी भी यह नहीं बताता कि "क्या महत्वपूर्ण है"।
यह केवल उन शर्तों को संरक्षित करता है जिनके तहत वह प्रश्न उत्तरणीय बना रहता है।

---

## मूल सिद्धांत

> **एंट्रॉपी (अव्यवस्था) की अनुमति है। अपठनीयता नहीं।**

रेजिस्ट्रम वैश्विक स्तर पर एंट्रॉपी को कम नहीं करता है।
यह उन स्थानों को सीमित करता है जहां एंट्रॉपी मौजूद हो सकती है, ताकि समय के साथ पहचान, वंश और संरचना को जांचा जा सके।

---

## रेजिस्ट्रम कैसे काम करता है

### संरचनात्मक रजिस्ट्री

रजिस्ट्री एकमात्र संवैधानिक प्राधिकरण है:

- यह सभी अवस्था परिवर्तनों को 11 संरचनात्मक नियमों के विरुद्ध मान्य करता है।
- यह पहचान, वंश और क्रम संबंधी सीमाओं को लागू करता है।
- यह नियतात्मकता और पता लगाने की क्षमता की गारंटी देता है।
- यह उल्लंघनों को उजागर करता है लेकिन उन्हें हल नहीं करता (त्रुटि-रोधी)।

हर चीज इसी के माध्यम से पंजीकृत होती है। कुछ भी इससे नहीं गुजरता।

### 11 नियम

| वर्ग | गणना | उद्देश्य |
| ------- | ------- | --------- |
| पहचान | 3 | अद्वितीय, अपरिवर्तनीय, सामग्री-आधारित |
| वंश | 4 | वैध वंश, चक्रीय नहीं, पता लगाने योग्य |
| क्रम | 4 | एकदिष्ट, अंतराल-रहित, नियतात्मक |

ये नियम संवैधानिक हैं। इन्हें बदलने के लिए औपचारिक शासन की आवश्यकता होती है।

---

## दोहरे संवैधानिक साक्षी

रेजिस्ट्रम **दो स्वतंत्र नियम-जांच इंजन** बनाए रखता है:

| साक्षी | भूमिका | कार्यान्वयन |
| --------- | ------ | ---------------- |
| रजिस्ट्री | प्राथमिक प्राधिकरण | संकलित डीएसएल (आरपीजी v1) |
| विरासत | माध्यमिक साक्षी | टाइपस्क्रिप्ट विधेय |

फेज एच के अनुसार, **रजिस्ट्री डिफ़ॉल्ट संवैधानिक इंजन है।**
विरासत एक स्वतंत्र माध्यमिक साक्षी के रूप में बनी हुई है।

### दो साक्षियों की आवश्यकता क्यों है?

- **सहमति आवश्यक है** - एक अवस्था परिवर्तन को मान्य होने के लिए, दोनों को सहमत होना होगा।
- **असहमति से रोक** - असमानता प्रणाली को रोक देती है (त्रुटि-रोधी)।
- **स्वतंत्रता जानबूझकर है** - न तो कोई भी दूसरे को ओवरराइड कर सकता है।

यह एक सुरक्षा और पठनीयता सुविधा है, तकनीकी ऋण नहीं।

दोहरे-मोड स्थायी है। किसी भी साक्षी को हटाने की कोई योजना नहीं है।

### समानता प्रमाण

274 परीक्षण व्यवहारिक समानता को प्रमाणित करते हैं:
- सभी अपरिवर्तनीय वर्गों में 58 समानता परीक्षण
- 12 निरंतरता समानता परीक्षण (समय की स्थिरता)
- रीप्ले समानता: लाइव निष्पादन ≡ रीप्ले किया गया निष्पादन

---

## इतिहास, रीप्ले और ऑडिट क्षमता

### स्नैपशॉट

रजिस्ट्रम अपनी पूरी स्थिति का स्नैपशॉट ले सकता है:
- संस्करणित स्कीमा (`RegistrarSnapshotV1`)
- सामग्री-आधारित हैश
- नियतात्मक क्रमबद्धता

### रीप्ले

ऐतिहासिक निर्णयों को रीप्ले किया जा सकता है:
- केवल-पढ़ने योग्य निष्पादन एक नए रजिस्ट्रार के खिलाफ
- समय की नियतात्मकता को प्रमाणित करता है
- समान परिवर्तन → समान परिणाम

### ऑडिट क्षमता

प्रत्येक संरचनात्मक निर्णय:
- तथ्य के बाद पुन: प्रस्तुत किया जा सकता है
- निष्पादन संदर्भ से स्वतंत्र
- किसी भी पार्टी द्वारा स्नैपशॉट के साथ सत्यापित किया जा सकता है

## बाहरी प्रमाण (वैकल्पिक)

रजिस्ट्रम वैकल्पिक रूप से सार्वजनिक रूप से देखने के लिए एक बाहरी अपरिवर्तनीय लेज़र (जैसे XRPL) पर क्रिप्टोग्राफिक प्रमाण जारी कर सकता है।

| गुण | मान |
| ---------- | ------- |
| डिफ़ॉल्ट | अक्षम |
| अधिकार | गैर-अधिकार (केवल गवाह) |
| व्यवहार पर प्रभाव | कोई नहीं |

प्रमाणपत्र रिकॉर्ड करते हैं कि रजिस्ट्रम ने क्या निर्णय लिया।
रजिस्ट्रम यह तय करता है कि क्या मान्य है।

**अधिकार अंदर की ओर प्रवाहित होता है। गवाह बाहर की ओर प्रवाहित होता है।**

देखें:
- [`docs/WHY_XRPL.md`](docs/WHY_XRPL.md) — तर्क
- [`docs/ATTESTATION_SPEC.md`](docs/ATTESTATION_SPEC.md) — विनिर्देश

## शासन

रजिस्ट्रम एक **संवैधानिक मॉडल** के तहत शासित है।

| सिद्धांत | अर्थ |
| ----------- | --------- |
| व्यवहारिक गारंटी > सुविधा वेग | सटीकता को प्राथमिकता |
| केवल साक्ष्य-आधारित परिवर्तन | बिना प्रमाण के कोई परिवर्तन नहीं |
| औपचारिक प्रक्रिया आवश्यक | प्रस्ताव, कलाकृतियाँ, निर्णय |

### वर्तमान स्थिति

- **फेज एच**: पूर्ण (रजिस्ट्री डिफ़ॉल्ट, प्रमाण सक्षम)
- **शासन**: सक्रिय और लागू
- **सभी परिवर्तन**: औपचारिक शासन प्रक्रिया की आवश्यकता होती है

भविष्य में सभी परिवर्तन शासन निर्णय हैं, इंजीनियरिंग कार्य नहीं।

देखें:
- [`docs/governance/PHILOSOPHY.md`](docs/governance/PHILOSOPHY.md) — शासन क्यों मौजूद है
- [`docs/governance/SCOPE.md`](docs/governance/SCOPE.md) — क्या शासित है
- [`docs/GOVERNANCE_HANDOFF.md`](docs/GOVERNANCE_HANDOFF.md) — शासन में परिवर्तन

## शुरुआत कैसे करें

### स्थापना

```bash
npm install @mcp-tool-shop/registrum
```

### बुनियादी उपयोग

```typescript
import { StructuralRegistrar } from "@mcp-tool-shop/registrum";
import { loadCompiledRegistry } from "@mcp-tool-shop/registrum/registry";

// Option 1: Legacy mode (TypeScript predicates, no compiled registry needed)
const registrar = new StructuralRegistrar({ mode: "legacy" });

// Option 2: Registry mode (default — requires compiled registry)
// const compiledRegistry = loadCompiledRegistry();
// const registrar = new StructuralRegistrar({ compiledRegistry });

// Register a root state
const result = registrar.register({
  from: null,
  to: { id: "state-1", structure: { version: 1 }, data: {} }
});

if (result.kind === "accepted") {
  console.log(`Registered at index ${result.orderIndex}`);
} else {
  console.log(`Rejected: ${result.violations.map(v => v.invariantId)}`);
}
```

### उदाहरण चलाएं

`examples/` निर्देशिका में दिए गए उदाहरण **स्थिर एपीआई नहीं, बल्कि प्रदर्शनकारी उदाहरण** हैं।

इनको चलाने के लिए [`tsx`](https://github.com/esbuild-kit/tsx) की आवश्यकता है:

```bash
# Run the refusal-as-success example
npm run example:refusal

# Or directly with npx
npx tsx examples/refusal-as-success.ts
```

**नोट:** उदाहरण `npx tsx` (या `npx ts-node` ईएसएम समर्थन के साथ) पर आधारित हैं। ये उत्पादन निर्भरताएं नहीं हैं - ये विकास/प्रदर्शन उपकरण हैं।

- शक्ति से अधिक संयम
